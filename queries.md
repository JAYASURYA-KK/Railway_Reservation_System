## 1. index.php — Login Queries

### Q1.1 — Select User by Email (Login Authentication)

```sql
SELECT * FROM USERS WHERE Email = '$email'
```

| Property | Value |
|---|---|
| **Type** | SELECT (READ) |
| **Table** | USERS |
| **Purpose** | Fetch user record by email during login |
| **Where Used** | `index.php` line ~48 |
| **Input** | `$email` (escaped via `mysqli_real_escape_string`) |
| **Output** | All columns of the matching user row |
| **Flow** | Result is checked with `mysqli_num_rows > 0`, then `password_verify()` is called on the hashed password |
| **Security** | Uses `mysqli_real_escape_string` — prevents basic SQL injection but prepared statements would be safer |

**Explanation:**  
When a user submits the login form, this query retrieves the full user record matching the provided email. The PHP code then uses `password_verify()` to compare the submitted plaintext password against the bcrypt hash stored in the `Password` column. If both email and password match, session variables (`user_id`, `role`, `email`) are set and the user is redirected to their respective dashboard.

## 2. signup.php — Registration Queries

### Q2.1 — Check Email Uniqueness

```sql
SELECT Email FROM USERS WHERE Email = '$email'
```

| Property | Value |
|---|---|
| **Type** | SELECT (READ) |
| **Table** | USERS |
| **Purpose** | Check if the email is already registered before creating a new account |
| **Where Used** | `signup.php` line ~43 |
| **Input** | `$email` (escaped) |
| **Output** | Email column if found, empty result set if not |
| **Logic** | If `mysqli_num_rows > 0`, the email already exists → show error |

**Explanation:**  
Before inserting a new user, this query checks whether the given email address already exists in the USERS table. ### Q2.2 — Insert New User

```sql
INSERT INTO USERS (Name, Email, Phone, Password, Role) 
VALUES ('$full_name', '$email', '$phone', '$hashed_password', 'user')
```

| Property | Value |
|---|---|
| **Type** | INSERT (CREATE) |
| **Table** | USERS |
| **Purpose** | Register a new user account |
| **Where Used** | `signup.php` line ~50 |
| **Input** | `$full_name`, `$email`, `$phone` (escaped); `$hashed_password` (bcrypt hash) |
| **Default** | Role is always set to `'user'` (admins cannot be created via signup) |
| **Auto-Increment** | `UserID` is generated by MySQL |

**Explanation:**

## 3. search.php — Train Search Queries

### Q3.1 — Get All Distinct Station Names

```sql
SELECT DISTINCT StationName FROM STATION ORDER BY StationName
```

| Property | Value |
|---|---|
| **Type** | SELECT (READ) |
| **Table** | STATION |
| **Purpose** | Populate the "From" and "To" dropdown menus on the search form |
| **Where Used** | `search.php` line ~15 |
| **Output** | Sorted list of all station names |
| **DISTINCT** | Ensures no duplicate station names appear (though StationName is already UNIQUE) |

**Explanation:**  
This query fetches all unique station names from the STATION table to populate the source and destination dropdown selectors. The result set is reused twice (using `mysqli_data_seek(0)`) — once for the "From" dropdown and once for the "To" dropdown. Ordering alphabetically makes it easier for users to find their desired station.

---

### Q3.2 — Search Trains with Optional Filters

```sql
SELECT * FROM TRAIN 
WHERE FromStation = '$src' AND ToStation = '$dst' 
ORDER BY TrainNumber
```

| Property | Value |
|---|---|
| **Type** | SELECT (READ) — Dynamic Query |
| **Table** | TRAIN |
| **Purpose** | Search for trains matching source/destination filters |
| **Where Used** | `search.php` line ~20-24 |
| **Dynamic WHERE** | Filters are appended conditionally: only `FromStation`, only `ToStation`, both, or neither |
| **Filter Logic** | If no filters → `SELECT * FROM TRAIN ORDER BY TrainNumber` (returns all trains) |

**Filter Combinations:**

| Source | Destination | Query |
|---|---|---|
| Empty | Empty | `SELECT * FROM TRAIN ORDER BY TrainNumber` |
| Set | Empty | `SELECT * FROM TRAIN WHERE FromStation = '$src' ORDER BY TrainNumber` |
| Empty | Set | `SELECT * FROM TRAIN WHERE ToStation = '$dst' ORDER BY TrainNumber` |
| Set | Set | `SELECT * FROM TRAIN WHERE FromStation = '$src' AND ToStation = '$dst' ORDER BY TrainNumber` |

**Explanation:**  
This is a dynamically constructed query. The PHP code builds an array of filter conditions and joins them with `AND`. If no filters are provided, it returns all trains. The `FromStation` and `ToStation` columns in TRAIN are VARCHAR strings that reference STATION.StationName via foreign keys, so the comparison is string-based on station names (not IDs).

### Q4.2 — Count Confirmed Tickets (Seat Availability Check)

```sql
SELECT COUNT(*) as booked FROM TICKET WHERE TrainID = $train_id AND Status = 'CNF'
```

| Property | Value |
|---|---|
| **Type** | SELECT (READ) — Aggregate |
| **Table** | TICKET |
| **Purpose** | Count how many confirmed tickets exist for a train to check seat availability |
| **Where Used** | `booking.php` line ~31 |
| **Aggregate** | `COUNT(*)` returns the total number of confirmed bookings |
| **Logic** | If `booked_count < train.TotalSeats` → status = 'CNF'; else → status = 'WTL' |

**Explanation:**  
Before creating a ticket, the system must determine whether seats are still available. This query counts all tickets with status 'CNF' (Confirmed) for the given train. If the count is less than the train's TotalSeats capacity, the new ticket gets status 'CNF' and the fare is charged. Otherwise, the ticket goes to 'WTL' (Waitlist) with fare = 0. The `$booked_count` variable is incremented inside the loop for each passenger being booked in the same request.

---

### Q4.3 — Insert Ticket (Booking)

```sql
INSERT INTO TICKET (UserID, TrainID, PassengerID, Status, BookingDate, Fare) 
VALUES ($user_id, $train_id, $pid, '$status', '$booking_date', $fare)
```

| Property | Value |
|---|---|
| **Type** | INSERT (CREATE) |
| **Table** | TICKET |
| **Purpose** | Create a new ticket/booking for a passenger on a train |
| **Where Used** | `booking.php` line ~37 |
| **Input** | `$user_id` (from session), `$train_id`, `$pid` (passenger ID), `$status` ('CNF' or 'WTL'), `$booking_date` (today's date), `$fare` |
| **Auto-Increment** | `TicketID` generated by MySQL |
| **Auto-Generated** | `CreatedAt` defaults to `CURRENT_TIMESTAMP` |
| **Note** | `Pnr` column is left NULL (not generated in this flow) |

**Explanation:**  
This is the core booking query. It creates a TICKET record linking a user, a train, and a passenger. The Status is dynamically set based on seat availability: 'CNF' if seats are available, 'WTL' if the train is full. The Fare is set to the train's fare for confirmed tickets, or 0 for waitlisted tickets. This query runs inside a `foreach` loop, once per selected passenger.

## 5. bookings.php — User Bookings Queries

### Q5.1 — Get User's Bookings with JOIN

```sql
SELECT t.TicketID, p.PassengerName, tr.TrainNumber, tr.TrainName, tr.FromStation, 
       tr.ToStation, tr.DepartureTime, tr.ArrivalTime, t.Status, t.BookingDate, t.Fare
FROM TICKET t
JOIN PASSENGER p ON t.PassengerID = p.PassengerID
JOIN TRAIN tr ON t.TrainID = tr.TrainID
WHERE t.UserID = $user_id
ORDER BY t.BookingDate DESC
```

| Property | Value |
|---|---|
| **Type** | SELECT (READ) — Multi-table JOIN |
| **Tables** | TICKET t, PASSENGER p, TRAIN tr |
| **Purpose** | Fetch all bookings for the logged-in user with passenger and train details |
| **Where Used** | `bookings.php` line ~17 |
| **Joins** | `TICKET ⟶ PASSENGER` (on PassengerID), `TICKET ⟶ TRAIN` (on TrainID) |
| **Sort** | By BookingDate descending (newest first) |

**Explanation:**  
This is the most complex read query in the user-facing pages. It joins three tables to create a complete booking view: TICKET provides the booking details (status, date, fare), PASSENGER provides the traveler's name, and TRAIN provides the route and timing information. The WHERE clause filters to only show bookings belonging to the current logged-in user. Results are sorted newest-first so the user sees their most recent bookings at the top.

**JOIN Diagram:**
```
TICKET (t)
  ├── JOIN PASSENGER (p)  ON t.PassengerID = p.PassengerID
  └── JOIN TRAIN (tr)     ON t.TrainID = tr.TrainID
```

## 10. admin_status.php — Train Status Queries

### Q10.1 — Update All Tickets for a Train

```sql

```

**Explanation:**

### Q10.3 — Get Status History (Derived from Tickets)

```sql
SELECT tr.TrainNumber, t.Status, MAX(t.CreatedAt) AS UpdatedAt 
FROM TICKET t 
JOIN TRAIN tr ON t.TrainID = tr.TrainID 
GROUP BY tr.TrainNumber, t.Status 
ORDER BY UpdatedAt DESC 
LIMIT 50
```

| Property | Value |
|---|---|
| **Type** | SELECT (READ) — Aggregate + JOIN + GROUP BY |
| **Tables** | TICKET t, TRAIN tr |
| **Purpose** | Show recent status update history per train |
| **Aggregate** | `MAX(t.CreatedAt)` gives the most recent update time for each train/status combination |
| **GROUP BY** | Groups by TrainNumber and Status |
| **Limit** | 50 most recent records |

**Explanation:**  
This query derives status history from the TICKET table. It groups tickets by train and status, showing the most recent timestamp for each combination. This gives the admin a view of when statuses were last changed for each train.

### Q12.2 — Delete User

```sql
DELETE FROM USERS WHERE UserID=$uid
```

| Property | Value |
|---|---|
| **Type** | DELETE (REMOVE) |
| **Table** | USERS |
| **Cascade** | All PASSENGER and TICKET records for this user will be CASCADE deleted |
| **Warning** | Highly destructive — removes the user AND all their passengers AND all their tickets |

**Explanation:**  
Deleting a user triggers two levels of CASCADE:  
1. PASSENGER rows with this UserID are deleted (ON DELETE CASCADE from PASSENGER)  
2. TICKET rows with this UserID are deleted (ON DELETE CASCADE from TICKET)  
Additionally, TICKET rows referencing the deleted PASSENGER records are also cascade-deleted.  
This means deleting one user can remove records from three tables.

## 13. schema.sql — DDL & Seed Queries

### DDL (Data Definition Language)

#### Create Database
```sql
CREATE DATABASE IF NOT EXISTS train_booking;
USE train_booking;
```

#### Drop Tables (safe reset order — respects foreign keys)
```sql
DROP TABLE IF EXISTS TSTATUS;
DROP TABLE IF EXISTS TICKET;
DROP TABLE IF EXISTS PASSENGER;
DROP TABLE IF EXISTS TRAIN;
DROP TABLE IF EXISTS STATION;
DROP TABLE IF EXISTS USERS;
```

**Order matters:** Dependent tables are dropped first to avoid foreign key constraint errors.

#### Create Tables

| Table | Primary Key | Foreign Keys | Unique Constraints |
|---|---|---|---|
| USERS | UserID (AI) | — | Email |
| STATION | StationID (AI) | — | StationName |
| TRAIN | TrainID (AI) | FromStation → STATION.StationName, ToStation → STATION.StationName | TrainNumber |
| PASSENGER | PassengerID (AI) | UserID → USERS.UserID (CASCADE) | — |
| TICKET | TicketID (AI) | UserID → USERS.UserID (CASCADE), TrainID → TRAIN.TrainID (CASCADE), PassengerID → PASSENGER.PassengerID (CASCADE) | Pnr |
| TSTATUS | (TrainID, SDate, Class) | TrainID → TRAIN.TrainID (CASCADE) | — |

#### Create Indexes
```sql
CREATE INDEX idx_user_email ON USERS(Email);

```

| Index | Table | Column(s) | Purpose |

| idx_user_email | USERS | Email | Speed up login lookup |
